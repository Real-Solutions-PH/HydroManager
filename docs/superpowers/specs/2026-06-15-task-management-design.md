# SP2 — Task Management + "More" tab

**Date:** 2026-06-15
**Status:** Approved (design)
**Scope:** New backend `tasks` module (+ migration), a "More" tab, full task-management mobile UI, a unified "Today" daily view, and local push reminders.

## Goal

Give growers a comprehensive personal task manager — title, body, priority, due date, and recurrence — surfaced daily so it guides day-to-day hydroponics operations. Reached through a new **More** tab that also houses the existing Library and Settings.

## Decisions (from brainstorming)

- **More tab** → landing menu (cards: Tasks, Library, Settings).
- **Daily surfacing** → a unified **Today** view that merges user tasks due today with the existing auto-generated checklist (pH, water change, harvest-ready, …).
- **Reminders** → local OS notifications via `expo-notifications` (local only; web no-op; full device testing may require a dev build, not Expo Go).
- **Fields** → include priority (High/Medium/Low/None). Recurrence presets: daily / weekly / monthly, each with an "every N" interval.

## Environment

Expo SDK 52, RN 0.76, new arch on. Existing UI built on `@rn-primitives/*`, RN `StyleSheet` + theme constants, TanStack Query, Zustand, MMKV, `@react-native-community/datetimepicker`. Reuse these — do not add new UI primitive libraries.

## Phase A — Backend `tasks` module

Mirror the `batches` module layout (`models.py`, `schema.py`, `repo.py`, `services.py`, `routes.py`, `main.py`).

### Model — `Task` (table `task`)

| Field | Type | Notes |
|---|---|---|
| `id` | UUID PK | |
| `owner_id` | UUID FK `user.id` ondelete CASCADE | scoping |
| `title` | str(1..200) | |
| `body` | str(0..2000) \| None | |
| `priority` | enum `TaskPriority` {high, medium, low, none} | default `none` |
| `due_at` | datetime(tz) \| None | deadline; drives Today + reminders |
| `recur_freq` | enum `RecurFreq` {none, daily, weekly, monthly} | default `none` |
| `recur_interval` | int ≥ 1 | default 1 — "every N" units |
| `completed_at` | datetime(tz) \| None | set for completed non-recurring tasks |
| `last_completed_at` | datetime(tz) \| None | set each time a recurring task is completed |
| `created_at` | datetime(tz) | |
| `updated_at` | datetime(tz) | bumped on update/complete |

### Service rules

- `_authorize` by `owner_id` (superuser bypass), as in batches.
- `complete_task`:
  - `recur_freq == none` → set `completed_at = now`, bump `updated_at`.
  - recurring → set `last_completed_at = now`; advance `due_at` to the next future occurrence via `_next_due(due_at, freq, interval)` (repeatedly add `interval` × freq-unit until `> now`, so missed runs are skipped, not piled up); leave `completed_at` null. If `due_at` is null, completing just stamps `last_completed_at`.
- `uncomplete_task`: clear `completed_at` (non-recurring), bump `updated_at`.
- `_next_due`: daily → `+interval days`; weekly → `+interval weeks`; monthly → add `interval` months (clamp day-of-month overflow to month end).

### Endpoints

- `GET /tasks/` — query: `include_completed` (default false), `due_before` (ISO), `skip`, `limit`. Returns `{ data, count }`, ordered by `due_at` asc nulls last, then `priority`.
- `POST /tasks/` — create.
- `GET /tasks/{id}`.
- `PATCH /tasks/{id}` — partial update.
- `POST /tasks/{id}/complete`, `POST /tasks/{id}/uncomplete`.
- `DELETE /tasks/{id}`.

### Schemas

`TaskBase`, `TaskCreate`, `TaskUpdate` (all optional), `TaskPublic`, `TasksPublic` ({data, count}).

### Wiring

- Import `Task` in `app/db/models.py`.
- Include `tasks_router` in `app/api/v1.py`.
- Alembic migration `add task table` (autogenerated, reviewed). Apply to local test DB before tests.
- No activity-log integration in v1 (avoids touching the `ActivityType` enum).

### Tests — `backend/tests/api/routes/test_tasks.py`

Fresh user per test (free tier safe — tasks have no quota). Cover:
- create → appears in list; defaults (`priority=none`, `recur_freq=none`).
- complete non-recurring → `completed_at` set; excluded from default list, present with `include_completed=true`.
- complete recurring (daily, interval 2) → `due_at` advances by 2 days, stays in default list, `last_completed_at` set.
- `_next_due` skips missed runs: due_at in the past by several intervals → advances to a future time.
- update fields (PATCH) persists.
- delete removes it.
- owner isolation: a second user cannot GET/PATCH/DELETE another's task (403/404).

## Phase B — More tab + Tasks CRUD UI

### Navigation

- Add `{ key: "more", label: "More", icon: Menu }` to `TAB_ITEMS` in `app/(app)/_layout.tsx` (6 items — confirm `InteractiveMenu` renders them). Register hidden tabs for `more`-visible and the new `tasks/*` routes.
- `app/(app)/more.tsx` — landing menu; cards linking to `/tasks`, `/library`, `/settings`.
- `app/(app)/tasks/index.tsx`, `app/(app)/tasks/new.tsx`, `app/(app)/tasks/[id].tsx`.

### Data layer

- Run `openapi-ts` to regenerate `src/client`.
- Add `tasksApi` to `src/lib/hydro-api.ts` (wrapper matching `setupsApi`/`batchesApi`): `list`, `get`, `create`, `update`, `complete`, `uncomplete`, `delete`. Export `Task`, `TaskPriority`, `RecurFreq`, `TaskCreate`, `TaskUpdate` types.
- Add `QK.tasks` keys + `STALE.tasks` in `src/lib/query-config.ts`.

### Tasks list (`tasks/index.tsx`)

- Segmented control **Today | All** (reuse existing tabs/segmented primitive).
- **All:** user tasks grouped by priority (High / Medium / Low / None) into collapsible sections (matching the TickTick reference), each row showing title, due-date badge, recurrence icon. Tap row → edit. Checkbox completes (optimistic via `tasksApi.complete`). A collapsed "Completed" section at the bottom.
- FAB `+` → `tasks/new`.
- Optimistic updates using the existing `lib/optimistic.ts` helpers + `useCustomToast`.

### Task form (`tasks/new.tsx`, `tasks/[id].tsx`)

Shared form component: title (Input), body (multiline Input), priority chips, due date + time (`DatePicker` + time via datetimepicker), recurrence (freq chips none/daily/weekly/monthly + interval stepper shown when freq ≠ none). Save → create/update; delete on edit screen. Reuse existing ui components only.

## Phase C — Unified Today view

- Extract the existing auto-checklist derivation (`deriveLocalTasks`, `checklistApi` query, MMKV per-day completion in `app/(app)/checklist.tsx`) into a reusable hook `src/hooks/use-auto-checklist.ts`.
- **Today** tab renders, in one list: overdue + due-today user tasks (checkable → `tasksApi.complete`) and auto-checklist items (checkable → existing MMKV toggle). Sort: overdue first, then by priority, then due time. Empty state when nothing due.
- The standalone `checklist` screen keeps working via the same hook (no behavior change); the Home quick-link to it remains.

## Phase D — Local notifications

- Add `expo-notifications`; add to `app.json` `plugins`.
- `src/lib/notifications.ts`:
  - `ensurePermission()` — request once; remember outcome in MMKV.
  - `syncTaskReminders(tasks)` — cancel previously app-scheduled task notifications, then schedule one local notification at each future, uncompleted `due_at` (title = task title, body = task body/priority). Persist a `taskId → notificationId` map in MMKV.
  - All functions guarded by `Platform.OS !== "web"` (no-op on web).
- Set a foreground notification handler in `src/components/providers.tsx`.
- Call `syncTaskReminders` after the tasks query resolves and after create/update/complete/delete mutations settle.
- Request permission the first time a user saves a task that has a `due_at`.

## Out of scope

- Linking a task to a specific setup/batch.
- Weekly-on-specific-weekdays recurrence (only "every N weeks" from the due weekday).
- Remote/push notifications; shared or team tasks.
- Replacing the existing checklist screen (kept alongside the unified Today view).
